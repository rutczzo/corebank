# Withdraw Concurrency Comparison Summary

## 실험 목적

Spring Boot + PostgreSQL 기반 Core Banking 프로젝트에서 동일 계좌 동시 출금 시나리오를 대상으로
`PESSIMISTIC_WRITE` 적용 여부에 따른 차이를 `k6` 기준 클라이언트 관점으로 비교했습니다.

비교 대상은 아래 두 상태입니다.

- 락 적용 상태
- 락 해제 상태

이번 실험은 운영 메트릭 계측이 아니라, 동일 조건에서 처리량과 응답시간, 상태코드 분포, 최종 정합성 차이를 확인하는 목적입니다.

## 실험 조건

- API: `POST /api/transactions/withdraw`
- 계좌: `A-0001`
- 요청 금액: `1000`
- 초기 잔액: `100000`
- 동시 요청 수: `150`
- k6 시나리오: `shared-iterations`, `150 VUs`, `150 iterations`

## 비교 결과

| 항목 | 락 미적용 | 락 적용 |
|---|---:|---:|
| 처리량(req/sec) | 148.12 | 103.28 |
| 평균 응답시간(ms) | 627.36 | 924.06 |
| p95(ms) | 870.07 | 1298.04 |
| HTTP 200 | 150 | 100 |
| HTTP 422 | 0 | 50 |
| 성공 거래 수 | 150 | 100 |
| 최종 잔액(`A-0001`) | 84,000 | 0 |
| 정합성 | 불일치 발생 | 정상 |

## 해석

락을 제거하면 처리량은 더 높고 응답시간도 더 짧게 보입니다.
하지만 `150건` 모두 `200 OK`를 반환했음에도 최종 잔액이 `84,000원`으로 남아,
응답 결과와 실제 데이터 상태가 일치하지 않는 lost update가 발생했습니다.

반대로 `PESSIMISTIC_WRITE`를 적용하면 처리량은 낮아지고 p95도 증가합니다.
대신 실제로 처리 가능한 `100건`만 성공시키고, 나머지 `50건`은 `422`로 실패 처리하면서
최종 잔액을 `0원`으로 유지해 정합성을 보장했습니다.

즉, 금융 거래에서는 락이 없는 쪽이 더 빠른 것이 아니라,
잘못된 결과를 더 빨리 반환하는 상태일 수 있다는 점을 확인할 수 있습니다.

## 포트폴리오/자소서용 요약

Spring Boot + PostgreSQL 기반 Core Banking 시뮬레이션에서 동일 계좌 동시 출금 시나리오를 `k6`로 비교 실험했습니다.
비관적 락을 제거한 경우 처리량은 높아졌지만, `150건` 모두 `200 OK`를 반환했음에도 최종 잔액이 `84,000원`으로 남아 정합성 불일치가 발생했습니다.
반대로 `PESSIMISTIC_WRITE`를 적용한 경우 처리량은 낮아졌지만 `100건 성공, 50건 422`, 최종 잔액 `0원`으로 데이터 정합성을 유지했습니다.
이를 통해 금융 도메인에서는 단순 처리량보다 동시성 제어와 일관성 보장이 우선이라는 점을 실험으로 입증했습니다.

## 면접 답변용 요약

동일 계좌에 `150건`의 동시 출금을 보내는 `k6` 실험으로 비관적 락 유무를 비교했습니다.
락을 제거하면 처리량은 약 `148 req/sec`로 더 높고 응답시간도 더 짧았지만, 모든 요청이 `200`이어도 최종 잔액이 `84,000원`으로 남아서 lost update가 발생했습니다.
반면 `PESSIMISTIC_WRITE`를 적용하면 처리량은 약 `103 req/sec`로 줄고 p95도 늘었지만, `100건만 성공`하고 나머지 `50건은 422`로 실패 처리되면서 최종 잔액은 정확히 `0원`이 됐습니다.
즉, 금융 거래에서는 성능 수치만 보면 락 없는 쪽이 좋아 보여도 실제로는 잘못된 결과를 더 빨리 내는 상태일 수 있습니다.
이 실험을 통해 동시성 제어는 단순한 성능 저하 요인이 아니라 정합성 확보 비용이라는 점을 설명할 수 있습니다.
